# 测试方案

## 一、测试策略总览

| 测试类型 | 覆盖范围 | 工具 |
|---------|---------|------|
| 单元测试 | 业务逻辑、AI 解析器、分类匹配 | Vitest |
| 集成测试 | API 路由 + 数据库 | Vitest + supertest + 内存 SQLite |
| E2E 测试 | 核心用户流程 | Playwright |
| 性能测试 | 接口响应时间 | k6 / autocannon |

## 二、单元测试用例

### 2.1 AI 解析模块

| 用例 | 输入 | 预期输出 |
|------|------|---------|
| 单条支出 | "午饭32" | `[{type:"expense", amount:3200, category:"餐饮"}]` |
| 多条批量 | "早餐8，午饭25，地铁4" | 3 条记录，金额 800/2500/400 |
| 收入 | "发工资12000" | `[{type:"income", amount:1200000, category:"工资"}]` |
| 转账 | "微信转支付宝500" | `[{type:"transfer", amount:50000, account:"微信", target_account:"支付宝"}]` |
| 金额小数 | "打车15.5" | amount = 1550 |
| 无金额 | "买了点东西" | 解析失败，返回 fallback |
| 日期推断 | "昨天午饭30" | date = 昨天日期 |
| 账户推断 | "微信付了咖啡28" | account = "微信" |
| 金额含单位 | "花了三十二块" | amount = 3200 |
| 空输入 | "" | 解析失败 |

### 2.2 分类匹配器

| 用例 | AI 返回值 | 预期匹配 |
|------|----------|---------|
| 精确匹配 | "餐饮" | → 餐饮 |
| 包含匹配 | "餐饮美食" | → 餐饮 |
| 反向包含 | "饭" | → 餐饮（餐饮包含"饭"？不，走同义词） |
| 同义词 | "吃饭" | → 餐饮 |
| 同义词 | "出行" | → 交通 |
| 同义词 | "买东西" | → 购物 |
| 无法匹配 | "量子计算" | → 其他 |
| 空字符串 | "" | → 其他 |

### 2.3 金额转换

| 输入（元） | 输出（分） | 说明 |
|-----------|-----------|------|
| 32 | 3200 | 整数 |
| 15.5 | 1550 | 一位小数 |
| 99.99 | 9999 | 两位小数 |
| 0.01 | 1 | 最小单位 |
| 0.1 + 0.2 | 30 | Math.round 防浮点 |
| 10000 | 1000000 | 大金额 |

### 2.4 幂等逻辑

| 用例 | 操作 | 预期 |
|------|------|------|
| 首次提交 | POST client_id=xxx | 创建成功 |
| 重复提交 | POST 相同 client_id | 返回已有记录，不重复入库 |
| 不同用户同 client_id | POST | 各自独立创建 |
| client_id 为空 | POST | 正常创建（不做幂等检查） |

### 2.5 认证模块

| 用例 | 操作 | 预期 |
|------|------|------|
| 有效邀请码注册 | code 有效且未用完 | 成功 + 默认分类/账户生成 |
| 过期邀请码 | expires_at < now | 拒绝 400 |
| 用完的邀请码 | used_count >= max_uses | 拒绝 400 |
| 重复用户名 | username 已存在 | 拒绝 400 |
| 正确密码登录 | bcrypt 匹配 | 返回 JWT |
| 错误密码登录 | bcrypt 不匹配 | 401 |
| 禁用用户登录 | is_active = 0 | 401 |
| 无 token 访问 | 不带 Authorization | 401 |
| 过期 token | JWT 已过期 | 401 |
| 伪造 token | 签名不匹配 | 401 |
| user 调 admin 接口 | role = user | 403 |

## 三、集成测试用例

### 3.1 交易 CRUD

| 用例 | 步骤 | 验证 |
|------|------|------|
| 创建单条支出 | POST items=[1条] | created[0] 存在，DB 有记录 |
| 创建批量 | POST items=[3条] | created.length = 3 |
| 转账记录 | type=transfer, target_account_id 必填 | category_id 为 null |
| 转账不计入统计 | 创建转账后查统计 | 收支数据不变 |
| 查询筛选-分类 | GET ?category_id=1 | 只返回该分类 |
| 查询筛选-日期 | GET ?start_date&end_date | 只返回范围内 |
| 查询筛选-关键词 | GET ?keyword=午饭 | description 包含"午饭" |
| 修改交易 | PUT 修改金额 | 金额更新，updated_at 更新 |
| 软删除 | DELETE | deleted_at 有值 |
| 删除后列表不显示 | DELETE 后 GET 列表 | 不包含已删除记录 |
| 删除后不计入统计 | DELETE 后 GET stats | 金额不含已删除 |
| 跨用户隔离 | user1 创建，user2 查询 | user2 看不到 |

### 3.2 统计接口

| 用例 | 前置数据 | 验证 |
|------|---------|------|
| 月度摘要 | 3笔支出 + 1笔收入 | expense/income/balance 正确 |
| 转账不计入 | 1笔转账 | 不影响收支 |
| pending 不计入 | 1笔 status=pending | 不出现在统计 |
| 软删除不计入 | 删除1笔后查统计 | 不包含已删除 |
| 分类排行 | 多分类支出 | 按金额降序 |
| 日趋势 | 跨3天数据 | 每天金额正确 |

### 3.3 预算

| 用例 | 步骤 | 验证 |
|------|------|------|
| 设置总预算 | POST category_id=0 | 创建成功 |
| 设置分类预算 | POST category_id=1 | 创建成功 |
| 使用率计算 | 预算10000分 + 支出8500分 | percent=85% |
| 唯一约束 | 同用户同分类同月重复 | 拒绝 |
| 超支状态 | 支出>预算 | status=exceeded |

## 四、E2E 测试用例

| 场景 | 步骤 | 验证点 |
|------|------|--------|
| 注册完整流程 | 输入邀请码+用户名+密码 → 提交 | 注册成功，进入引导页 |
| 引导→记账 | 引导完成 → AI输入"午饭32" → 确认 | 流水出现"午饭 ¥32.00" |
| AI 批量记账 | 输入"早餐8，午饭25，地铁4" → 展示3张卡片 → 全部确认 | 3笔入库 |
| 确认卡片编辑 | AI 解析后 → 修改金额 → 确认 | 以修改后金额入库 |
| 确认卡片删除单条 | 3条结果 → 删除1条 → 确认 | 2笔入库 |
| AI 失败降级 | mock AI 超时 → 弹出手动表单 | 表单预填原始输入 |
| 手动记账 | 填写表单 → 提交 | 记录入库，流水显示 |
| 编辑交易 | 点击流水 → 修改金额 → 保存 | 金额更新 |
| 删除交易 | 点击删除 → 确认 | 流水消失，统计更新 |
| 跨用户隔离 | user1 记账 → user2 登录 | user2 看不到 user1 数据 |
| 统计展示 | 记账后进入统计页 | 收支摘要/分类饼图正确 |
| 响应式-手机 | 375px 视口 | 底部 tab 导航，单列布局 |
| 响应式-PC | 1200px 视口 | 侧边栏导航，宽内容区 |
| 离线态 | 断网后打开 | 显示手动表单，隐藏 AI 入口 |

## 六、安全测试

| 用例 | 验证 |
|------|------|
| SQL 注入 | 筛选参数含注入字符 → 不影响系统 |
| JWT 伪造 | 修改 payload → 401 |
| JWT 过期 | 超 30 天 → 401 |
| 越权-角色 | user 调 /api/admin/* → 403 |
| 越权-数据 | 访问他人 transaction_id → 404 |
| AI Key 泄露 | 前端代码/响应中不含 API Key |
| XSS | description 含 `<script>` → 前端转义 |
| CSRF | 无 cookie 认证，JWT Bearer 天然防 CSRF |
| 密码存储 | 数据库中为 bcrypt hash，非明文 |
| 邀请码爆破 | 连续错误应有频率限制 |

## 七、AI 质量验收指标

| 指标 | 目标 | 计算公式 |
|------|------|---------|
| 一次确认率 | > 80% | confirm_no_edit / (no_edit + with_edit + cancel) |
| 修正率 | < 15% | confirm_with_edit / (no_edit + with_edit) |
| 失败率 | < 5% | ai_parse_fail / ai_parse_request |
| 响应 P95 | < 5s | ai_parse_success 的 latency_ms |
| 分类准确率 | > 85% | AI 推断分类 vs 用户最终确认分类 |

## 八、测试数据准备

### 8.1 固定测试数据

- 有效邀请码 3 个（max_uses=1/5/不限）
- 过期邀请码 1 个
- 用完邀请码 1 个
- 测试用户 2 个（user1 + user2，验证隔离）
- admin 用户 1 个

### 8.2 交易测试数据

- 50+ 条交易覆盖：
  - 支出/收入/转账
  - 多分类（餐饮/交通/购物/住房等）
  - 多账户（微信/支付宝/银行卡/现金）
  - 跨月数据（验证月度统计边界）
  - 含 pending 状态记录
  - 含 deleted_at 记录

### 8.3 Mock 数据

- Mock LLM 正常响应（JSON 数组）
- Mock LLM 超时（>10s）
- Mock LLM 非法 JSON（含 markdown 代码块包裹）
- Mock LLM 分类名不匹配
- Mock LLM 无金额响应

## 九、测试执行规范

### 9.1 运行方式

```bash
# 单元测试
cd server && npm run test

# 集成测试（使用内存 SQLite）
cd server && npm run test:integration

# E2E 测试
cd web && npx playwright test

# 性能测试
npx autocannon -c 10 -d 30 http://localhost:3000/api/transactions
```

### 9.2 CI 集成

- PR 合并前必须通过：单元测试 + 集成测试
- 每日定时运行：E2E 测试
- 每周运行：性能测试（对比基线）

### 9.3 覆盖率要求

| 模块 | 行覆盖率目标 |
|------|-------------|
| AI 解析器 | ≥ 90% |
| 分类匹配器 | ≥ 95% |
| 交易服务 | ≥ 85% |
| 认证模块 | ≥ 90% |
| 统计服务 | ≥ 80% |
| 整体 | ≥ 80% |
