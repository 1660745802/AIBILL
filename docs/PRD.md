# AI 记账 - 产品设计文档

## 1. 产品概述

### 1.1 产品名称
**AI 记账** (Bill)

### 1.2 定位
自部署的 **AI 个人财务工作台**，支持小范围多人使用。

不只是记账——从记录、统计、分析到规划（预算/负债/投资），AI 贯穿全链路。

首版核心主张：**最快把账记准**。后续逐步扩展为完整的财务管理平台。

### 1.3 目标用户

首批核心用户：**你自己 + 身边 3-5 个有记账意愿但嫌麻烦的朋友**。

| 画像 | 特征 | 核心需求 |
|------|------|---------|
| 主力用户（你） | 技术能力强，愿意折腾部署 | 全功能，数据掌控 |
| 日常用户（朋友） | 不想装 App、不想注册复杂账号 | 打开网页说句话就记完，偶尔看看花了多少 |
| 家庭成员（可选） | 不关心技术，只要能用 | 极简操作，能看月度账单 |

> 设计原则：满足"日常用户"的体验标准——如果他用着觉得麻烦，就是设计有问题。

### 1.4 核心体验承诺

"最快把账记准"拆解为可验证的体验指标：

| 指标 | 目标 | 验证方式 |
|------|------|---------|
| 打开→入账步数 | ≤ 3 步（输入→确认→完成） | 用户测试 |
| AI 成功时总耗时 | < 5 秒 | 从输入到确认卡片出现 |
| AI 失败时用户损失 | 0（无感切手动，输入内容保留） | 失败时预填到手动表单 |
| 批量记账效率 | 3 条 ≈ 1 条的时间 | "早餐8午饭25地铁4"一次确认 |

### 1.5 设计原则
- **记账零摩擦**：说一句话就记完，不用填表单
- **记得准**：AI 解析 + 用户确认，常见输入一次通过率 > 80%
- **数据在自己手里**：自部署，业务数据不上传第三方
- **多端统一接口**：Web、App 共用同一套 API，来源可追溯
- **每个用户完全独立**：各自记账互不可见，管理员不能看他人数据
- **AI 失败不阻塞**：任何 AI 环节出错，用户都能无感切到手动完成

### 1.6 本文档范围

本文档仅覆盖 **服务端 API + Web 前端（含 PWA）**。

| 端 | 本文档 | 说明 |
|----|--------|------|
| 服务端 | ✅ | Node.js + Fastify + SQLite，所有业务逻辑和 AI 调用 |
| Web 端 | ✅ | Vue3 + PWA，PC 和手机浏览器均可用 |
| App 端 (Android/iOS) | ❌ 另行设计 | 原生能力（通知监听、自动记账、推送等）需单独文档 |

> App 端后续设计时复用同一套服务端 API，客户端独立开发。

### 1.7 多端接入规范（API 契约）

所有客户端（Web、App、脚本导入）走同一套交易创建接口：

```json
POST /api/transactions
{
  "items": [{
    "client_id": "uuid-v4",          // 客户端生成，幂等键（防重复提交/重试）
    "client_type": "web",            // web | app_android | app_ios | import_script
    "client_created_at": "ISO8601",  // 客户端本地生成时间（离线场景用）
    "source": "ai",                  // manual | ai | import_csv | app_notification | ocr | subscription(P2)
    "source_detail": "...",          // 原始来源内容（通知文本/CSV行/OCR文本等）
    "type": "expense",
    "amount": 3200,
    "category_id": 1,                // transfer 类型为 null
    "account_id": 1,
    "target_account_id": null,       // 仅 transfer 时必填
    "description": "午饭",
    "date": "2026-07-01"
  }]
}
```

> **统一规则**：始终使用 items 数组格式，单条记账传 `items: [单条对象]`，批量传多条。

| 字段 | 必填 | 说明 |
|------|------|------|
| client_id | ✅ | UUID v4，相同 client_id 的重复请求幂等处理（返回已有记录） |
| client_type | ✅ | 标识来源客户端类型 |
| client_created_at | 否 | 离线记账时的客户端本地时间 |
| source | ✅ | 入账方式 |
| source_detail | 否 | 原始来源文本，用于追溯和去重 |

> **幂等规则**：同一 user_id + client_id 的请求，第二次直接返回已有记录，不重复入库。

| 层级 | 技术 | 理由 |

### 2.3 认证方案

采用 **邀请码注册 + JWT 登录** 方案，支持小范围多人使用。

#### 角色
| 角色 | 权限 |
|------|------|
| admin | 管理邀请码、管理用户、全局 AI 配置、自己的记账数据 |
| user | 注册/登录、自己的记账数据 |

#### 权限边界（隐私原则）

记账数据极度私密，**管理员不可查看其他用户数据**。

| admin 能做 | admin 不能做 |
|------------|-------------|
| 生成/作废邀请码 | 查看其他用户的交易记录 |
| 启用/禁用用户账号 | 修改其他用户的数据 |
| 配置全局共享 AI Key | 查看其他用户的 AI 对话 |
| 查看系统概况（用户数、总笔数） | 查看任何具体金额/内容 |

> 如果后续需要"家人共享账本"，应作为独立功能由双方主动授权，而非管理员默认可见。

#### 注册流程
```
用户访问注册页
  → 输入：邀请码 + 用户名 + 密码
  → 后端校验邀请码（是否存在、未过期、未用完）
  → 同一事务内：创建用户 + 生成默认分类 + 生成默认账户
  → 邀请码 used_count + 1
  → 返回 JWT
```

> **决策**：默认分类/账户在注册事务中由后端静默完成，前端无需额外调用。
> 首次引导仅做 UI 层引导（选择偏好、试记一笔），不负责数据初始化。

#### 登录流程
```
用户输入用户名 + 密码
  → 后端验证（bcrypt 比对 password_hash）
  → 签发 JWT（payload: { userId, role }, 有效期 30 天）
  → 前端存 localStorage，后续请求带 Authorization: Bearer <jwt>
```

> JWT 有效期设为 30 天——自部署小范围使用，安全性要求不高，减少频繁登录。

#### 数据隔离
- 所有业务表加 `user_id` 字段
- 每次查询/写入自动附加 `WHERE user_id = ?`
- 用户只能看到自己的数据
- **AI 问答上下文只注入当前用户的交易摘要**，绝不跨用户

#### 用户模型边界
- 每个用户完全独立记账，不存在"共享账本"或"成员权限"概念
- 注册成功后，系统自动为该用户生成默认分类和默认账户
- 用户可自行增删改分类/账户，互不影响
- 如果后续需要"家庭共享账本"，应作为独立功能另行设计，不在当前范围内

#### 邀请码管理
- admin 通过 API/设置页生成邀请码
- 可设置：最大使用次数（默认 1）、过期时间（可选）
- 用完自动失效

#### 配置
```env
# .env
JWT_SECRET=你的随机长字符串
ADMIN_USERNAME=admin
ADMIN_PASSWORD=初始管理员密码(首次启动自动创建)
```

#### API 认证中间件
```
请求进入
  → 提取 Authorization header
  → 验证 JWT 签名和有效期
  → 解析 userId、role 注入 request 上下文
  → 无效/过期 → 401
```

### 2.4 LLM 调用架构

**为什么放服务端：**
1. API Key 安全——不暴露在前端代码/浏览器中
2. 服务端拼接 prompt + 附带用户历史数据作为上下文
3. 换模型、调参数、加 fallback 都在一处改

**调用时序：**
```
前端                          后端                         LLM API
 │                             │                             │
 │  POST /api/ai/parse         │                             │
 │  { input: "午饭32" }        │                             │
 │ ──────────────────────────► │                             │
 │                             │  拼接 system prompt          │
 │                             │  + few-shot 示例             │
 │                             │  + 用户输入                  │
 │                             │                             │
 │                             │  POST /chat/completions     │
 │                             │ ──────────────────────────► │
 │                             │                             │
 │                             │  ◄─── JSON response ──────  │
 │                             │                             │
 │                             │  校验 JSON 格式              │
 │                             │  分类名 → category_id 匹配  │
 │                             │  补充默认值                  │
 │                             │                             │
 │  ◄── 结构化数据 ──────────  │                             │
 │                             │                             │
 │  展示确认卡片               │                             │
 │  用户点确认                 │                             │
 │                             │                             │
 │  POST /api/transactions     │                             │
 │ ──────────────────────────► │                             │
 │                             │  写入数据库                  │
 │  ◄── 成功 ─────────────── │                             │
```

**错误处理：**
| 异常场景 | 处理方式 |
|---------|---------|
| LLM 返回非法 JSON | 尝试修复（提取 JSON 片段）→ 失败则提示用户手动记 |
| LLM 超时（>10s） | 返回超时提示，提供"切换手动模式"按钮 |
| 分类名匹配不上 | 模糊匹配 → 匹配不上则归入"其他" |
| API Key 无效/余额不足 | 返回具体错误信息，引导去设置页检查 |
| 网络不通 | 前端离线时直接展示手动记账表单 |

## 3. 产品契约

### 3.1 统计口径

| 交易类型 | 计入支出 | 计入收入 | 计入预算 | 说明 |
|---------|---------|---------|---------|------|
| 普通支出 | ✅ | — | ✅ | 常规消费 |
| 普通收入 | — | ✅ | — | 工资、奖金等 |
| 转账 | ❌ | ❌ | ❌ | 账户间流转，不影响总收支 |
| 退款（收入分类） | — | ✅ | ❌ | 计入收入但不抵扣预算 |
| 报销（收入分类） | — | ✅ | ❌ | 同退款处理 |
| 信用卡还款 | ❌ | ❌ | ❌ | 建议记为转账（银行卡 → 信用卡） |

> **关于"报销"**：报销在系统中用"退款"分类或用户自建"报销"分类处理，统计口径等同退款——计入收入但不抵扣预算。

### 3.2 去重规则

防止同一笔交易从不同渠道重复入库：

| 场景 | 检测条件 | 处理 |
|------|---------|------|
| 幂等重试 | 同一 client_id | 直接返回已有记录 |
| 导入重复 | 同一 user + source + amount + date + description | 标记为"疑似重复"，提示用户确认 |
| App 通知重复 | 同一 source_detail 哈希 | 自动跳过 |

> P0 阶段只实现 client_id 幂等；"疑似重复"检测放 P1 和导入功能一起做。

### 3.3 交易确认状态

为 App 自动记账预留，P0 只用 `confirmed`：

| 状态 | 说明 | 场景 |
|------|------|------|
| confirmed | 用户已确认，正式入账 | 手动记账、AI确认后 |
| pending | 待确认（未来用） | App 自动识别的候选交易 |

> P0 所有交易直接是 confirmed。App 后续接入时，通知/OCR 自动识别的结果先进 pending，
> 用户在 App 或 Web 确认后变为 confirmed。pending 不计入统计。

### 4.1 P0 - 核心功能（第一版必须有）

#### 4.1.1 AI 快速记账
**输入方式**：
- 文字输入："今天午饭花了32"、"打车15.5"、"工资到账12000"
- 多条批量："早餐8，午饭25，地铁4"

**AI 解析流程**：
```
用户输入
  → 服务端拼接 prompt（含 few-shot 示例）
  → 调用 LLM
  → 解析响应 JSON
  → 校验格式（必须有 type + amount）
  → 分类名模糊匹配 → category_id
  → 金额转为整数（分）
  → 返回前端展示确认卡片
  → 用户确认/修改
  → 写入数据库
```

**失败回退**：任何环节失败 → 弹出手动记账表单，预填已解析的部分字段。

**解析结果结构（LLM 输出）**：
```json
[{
  "type": "expense",
  "amount": 32.00,
  "category": "餐饮",
  "description": "午饭",
  "date": "2026-07-01",
  "account": "微信"
}]
```

**服务端处理后返回前端**：
```json
[{
  "type": "expense",
  "amount": 3200,
  "category_id": 1,
  "category_name": "餐饮",
  "category_icon": "🍜",
  "description": "午饭",
  "date": "2026-07-01",
  "account_id": 1,
  "account_name": "微信"
}]
```

#### 确认卡片交互（P0 核心体验）

AI 解析后展示可编辑的确认卡片，用户可快速确认或微调：

```
┌─────────────────────────────────────────┐
│  ✅ 解析结果（2 条）                     │
├─────────────────────────────────────────┤
│  ┌─────────────────────────────────┐    │
│  │ 🍜 餐饮     ¥32.00     微信    │    │
│  │ 午饭        2026-07-01         │    │
│  │        [编辑] [删除]           │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │ 🚗 交通     ¥15.00     支付宝  │    │
│  │ 打车回家    2026-07-01         │    │
│  │        [编辑] [删除]           │    │
│  └─────────────────────────────────┘    │
│                                         │
│       [全部确认]    [取消]              │
└─────────────────────────────────────────┘
```

**交互规则**：
| 操作 | 说明 |
|------|------|
| 点击卡片任意字段 | 进入行内编辑（金额、分类下拉、账户下拉、日期选择器） |
| 删除单条 | 多条解析结果中移除不需要的 |
| 全部确认 | 批量写入，返回首页并刷新流水 |
| 取消 | 放弃本次 AI 结果，回到输入状态 |
| 编辑 | 展开完整表单（同手动记账表单），预填 AI 解析值 |

**AI 解析质量指标**（产品侧关注）：
| 指标 | 目标 | 说明 |
|------|------|------|
| 一次确认率 | > 80% | 用户无需修改直接确认的比例 |
| 手动修正率 | < 15% | 需要改 1-2 个字段的比例 |
| 失败率 | < 5% | AI 完全无法解析需切手动的比例 |

#### 确认卡片异常场景

| 场景 | 用户看到什么 | 处理方式 |
|------|------------|---------|
| 金额缺失 | 卡片金额栏高亮红色，显示"请填写金额" | 金额为空时"确认"按钮置灰 |
| 分类不确定 | 分类显示为"❓ 未分类"，底色浅黄 | 用户点击可下拉选择，或确认时归入"其他" |
| 账户为空 | 显示用户的默认账户（灰色，表示推断） | 用户可改，不改则入库时写 default_account_id |
| 日期歧义（"前天"跨月时） | 显示推断日期 + 小字"AI推断" | 用户可点击日期修改 |
| 多条中有一条解析失败 | 成功的正常展示，失败的显示为空白卡片+"手动补充" | 用户填完空白卡片后一起确认 |
| 全部解析失败 | "AI 没能理解，帮你打开手动记账" | 自动切换手动表单，输入内容保留为备注 |

#### 埋点定义

| 事件名 | 触发时机 | 关键参数 |
|--------|---------|---------|
| `ai_parse_request` | 用户提交 AI 记账 | input_length, item_count |
| `ai_parse_success` | AI 成功返回结构化数据 | item_count, latency_ms |
| `ai_parse_fail` | AI 返回失败/超时 | error_type, latency_ms |
| `confirm_no_edit` | 用户直接确认未修改 | item_count |
| `confirm_with_edit` | 用户修改后确认 | edited_fields[] |
| `confirm_cancel` | 用户取消 AI 结果 | — |
| `fallback_manual` | 切换到手动模式 | from_timeout \| from_error |

> 统计口径：
> - 一次确认率 = `confirm_no_edit` / (`confirm_no_edit` + `confirm_with_edit` + `confirm_cancel`)
> - 修正率 = `confirm_with_edit` / (`confirm_no_edit` + `confirm_with_edit`)
> - 失败率 = `ai_parse_fail` / `ai_parse_request`

## 6. AI 设计

### 6.1 记账解析 Prompt

```
你是一个记账助手，负责将用户的自然语言输入解析为结构化的记账数据。

规则：
1. 识别交易类型：支出(expense)、收入(income)、转账(transfer)
2. 提取金额（必须有，否则报错）
3. 推断分类（基于常见消费场景）
4. 提取日期（没有明确说则为今天：{today}）
5. 推断账户（没有明确说则为空字符串）
6. 支持一次输入多条记录，每条单独一个对象
7. transfer 类型的 category_id 为 NULL，不需要分类

可用分类（支出）：{user_expense_categories}
可用分类（收入）：{user_income_categories}
可用账户：{user_accounts}

## 示例

用户输入："午饭32，打车回家15块"
输出：
[
  {"type": "expense", "amount": 32, "category": "餐饮", "description": "午饭", "date": "{today}", "account": ""},
  {"type": "expense", "amount": 15, "category": "交通", "description": "打车回家", "date": "{today}", "account": ""}
]

用户输入："昨天发工资12000"
输出：
[
  {"type": "income", "amount": 12000, "category": "工资", "description": "发工资", "date": "{yesterday}", "account": ""}
]

用户输入："微信转支付宝500"
输出：
[
  {"type": "transfer", "amount": 500, "category": "", "description": "微信转支付宝", "date": "{today}", "account": "微信", "target_account": "支付宝"}
]

## 输出要求
只输出 JSON 数组，不要 markdown 代码块，不要任何其他文字。
```

> **转账交互定义**：
> - 转账是一条 `type: transfer` 记录，展示为"微信 → 支付宝 ¥500"
> - **transfer 类型的 category_id 为 NULL，不需要分类**
> - **不计入收入/支出统计，不计入预算消耗**
> - 仅影响两个账户的余额计算（来源账户减少，目标账户增加）
> - 编辑/删除只作用于这一条记录

你是一个个人财务分析助手。基于用户的交易数据回答财务相关问题。

## 7. 非功能需求

### 7.1 性能
- 首页加载 < 1s
- AI 记账响应 < 5s（含 LLM 调用，超时 10s 自动中断）
- 统计查询 < 500ms

### 7.2 安全与隐私

#### 接口安全
- 所有 API 需要 JWT 认证
- AI API Key 存在服务端，不发送到前端
- 用户间数据完全隔离，服务端强制按 user_id 过滤

#### 数据隐私原则

| 原则 | 说明 |
|------|------|
| 管理员不可见 | admin 不能查看任何其他用户的交易、余额、AI 对话 |
| AI 最小上传 | 只发送当前输入文本 + 必要摘要到 LLM，不发全量交易历史 |
| 软删除可恢复 | deleted_at 标记的记录保留 30 天，用户可在回收站恢复 |
| 硬删除有出口 | 用户可"永久删除"（清空 deleted 记录）或"注销账号"（清空全部数据） |
| 备份加密（P2） | 远程备份（WebDAV/OSS）建议加密传输和存储 |

#### AI 数据发送范围

| 场景 | 发送给 LLM 的内容 | 不发送 |
|------|------------------|--------|
| AI 记账 | 用户当前输入 + 分类/账户列表 | 历史交易数据 |
| AI 问答 | 当前月+上月摘要统计（聚合数字，token < 2000） | 逐条交易明细 |
| AI 信贷识别 | 单条交易描述 | 用户余额、其他交易 |

### 7.3 备份
- SQLite 单文件，cp 即备份
- 后续支持定时自动备份到指定目录
- 数据库 schema 变更通过 migration 管理

### 7.4 离线与弱网兜底

核心原则：**网络不好时记账不能断**。

#### P0（第一版必须有）

- AI 记账超时（>5s）→ 自动弹出手动记账表单，预填输入内容作为备注
- AI 问答超时 → 提示"网络不佳，稍后再试"
- Service Worker 缓存静态资源 → 离线也能打开页面
- 离线时隐藏 AI 入口，显示手动记账表单

#### P1（第二版）

| 能力 | 说明 |
|------|------|
| 本地暂存队列 | 离线时手动记账存入 IndexedDB，联网后自动同步 |
| 同步状态标记 | 待同步 ⏳ / 已同步 ✓ / 失败 ⚠️ |
| 前端数据缓存 | 分类/账户/最近流水缓存到 localStorage/IndexedDB |
| 弱网检测 | 顶部提示条"网络较慢" / "离线模式" |

#### P2（优化）

| 能力 | 说明 |
|------|------|
| Service Worker 拦截 POST | 记账请求失败自动存入队列 |
| 冲突处理 | 离线期间服务端有新数据时的合并策略 |
| 离线统计 | 基于本地缓存数据展示粗略统计 |

### 7.5 PWA 支持
- manifest.json：图标、名称、主题色、启动模式
- Service Worker：缓存静态资源
- 离线模式：可打开页面，显示手动记账表单（需联网才能调 AI 和同步）
